# DataRaum Cockpit — CLAUDE.md

TanStack Start web UI for the DataRaum engine. One of three packages in the [dataraum](https://github.com/dataraum/dataraum) monorepo (alongside `engine` + `infra`). The full architecture lives on the [DAT-339 pivot](https://real-dataraum.atlassian.net/wiki/spaces/DD/pages/23363586) Confluence page (the older "Cockpit + Engine REST" plan is superseded).

## Dev loop

The cockpit container in `packages/infra/docker-compose.yml` is for end-to-end smoke and prod-like serving. **For day-to-day UI work, run the dev server outside docker** — hot reload is much faster than rebuilding the container.

```bash
# Terminal 1 (workspace root, one-time)
docker compose -f packages/infra/docker-compose.yml up -d --wait control-plane postgres

# Terminal 2 (here)
bun install
bun run dev
# → http://localhost:3000 with hot reload

# Optional: if the engine added/changed SQLAlchemy models, refresh the
# Drizzle metadata client first:
#   DATARAUM_WORKSPACE_ID=<id> METADATA_DATABASE_URL=<url> bun run db:pull:metadata
```

The Vite dev server proxies `/api/*` to the engine on `:8000`, so the browser is same-origin and CORS is moot during dev.

## Playwright MCP — driving the UI from this session

Playwright is registered per-project in `~/.claude.json` under `projects['<repo path>'].mcpServers.playwright` (stdio, `npx @playwright/mcp@latest`) — not in a repo-local `.mcp.json`. The agent in this session gets browser tools (navigate / click / fill / snapshot / screenshot) automatically; nothing to install per-checkout. The cockpit dev server must be running on `:3000` for these tools to do anything useful.

Verify the registration:

```bash
python3 -c "import json; d=json.load(open('$HOME/.claude.json')); print(list(d['projects']['$PWD/../../'.rstrip('/')]['mcpServers'].keys()))"
```

(or just look for the `playwright` key under your repo path in `~/.claude.json`).

**Workflow when iterating on UI:**

1. Bring up the engine: `docker compose -f packages/infra/docker-compose.yml up -d` from workspace root
2. Bring up the cockpit: `bun run dev` from this directory
3. Tell the agent what to look at
4. Agent uses Playwright MCP to navigate, snapshot, identify issues, propose edits
5. Edits land hot via Vite; reload via Playwright; iterate

## Stack

- **TanStack Start** (React 19) — file-based routing + server functions
- **TanStack Router / Query** — type-safe routing + server state
- **Mantine v9** — component library
- **Tailwind CSS v4** — utility + layout alongside Mantine
- **Lucide React** — icons
- **Drizzle ORM** + `postgres` — cockpit_db (TS-side persistence)
- **Biome** — lint + format (no ESLint, no Prettier)
- **Vitest** — tests
- **TypeScript** strict, only

The full ecosystem (TanStack AI, xyflow, ECharts, CodeMirror, sql-formatter, marked, Apache Arrow JS) lands as widgets need it. See the [Tech Stack](https://real-dataraum.atlassian.net/wiki/spaces/DD/pages/18153474) page.

## Working with engine state

The engine exposes a 3-verb Starlette kernel (`/measure`, `/query`, `/probe`) plus `/health`. There is no OpenAPI spec and no codegen anymore — both retired in the DAT-339 pivot (Phase 0c deleted `packages/api/`, 0d deleted `src/api/`).

For metadata reads, the cockpit introspects the engine's `ws_<workspace_id>` Postgres schema directly via Drizzle: `bun run db:pull:metadata` produces typed clients in `src/db/metadata/`. Re-run after the engine adds or changes SQLAlchemy models.

Config data (vertical YAMLs etc.) lives in the `packages/dataraum-config/` sibling package and is bind-mounted read-only into the cockpit container at `DATARAUM_CONFIG_PATH` (`/opt/dataraum/config`) (DAT-361). No consumers yet — the mount is plumbing for future "which concept did the agent bind to?" UX hints, read via Node `fs`.

## Useful commands

```bash
bun run dev               # vite dev server
bun run build             # production build
bun run test              # vitest
bun run check             # biome check (lint + format)
bun run db:pull:metadata  # introspect ws_<workspace_id> from engine substrate → src/db/metadata/
bun run db:generate:cockpit  # cockpit_db migration SQL (from hand-written src/db/cockpit/schema.ts)
bun run db:push:cockpit   # push schema directly to cockpit_db
```

## Conventions

- Metadata reads go through the Drizzle metadata client (`src/db/metadata/`); the engine kernel verbs (`/measure`, `/query`, `/probe`) handle long-running operations.
- TanStack Start server functions for the chat surface (`/api/chat`) and any other BFF logic.
- One Mantine component per widget; Tailwind for layout/utility classes.
- `src/db/metadata/{schema,relations}.ts` is generated — never edit by hand. Re-run `bun run db:pull:metadata` after engine SQLAlchemy changes.
- Routes that need data should use a `loader: queryClient.ensureQueryData(...)` so SSR dehydrates the query.
