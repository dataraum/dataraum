Status: PLAN (full scope, 2026-06-24). The goal is not a minimal diff — it is to make BOTH

## Context model: don't cut, don't guess (settled 2026-06-24)

**The disease.** A cap (`MAX_SLICES`, `[:N]`) decides what to drop *without knowing what's
relevant* — a blind cut the agent can't even detect. Its mirror-image failure is a tool-loop
that fishes blindly, burns its turn budget, then answers on partial context. Both are blind.

**The two non-fixes.** (1) Cutting context is fixing a bug without the description. (2)
*Pre-selecting* "the relevant value-sets for this metric" is just a **smarter blind cut** —
nobody knows what's relevant up front; that's the agent's job at reason-time. So we neither
cut nor guess.

**The economics that let us not cut.** One LLM call **per metric**, looped (cap 10
concurrent), each carrying the grounding context. Input tokens cost ≈⅕ of output AND affect
latency AND quality (lost-in-the-middle). BUT: large windows (~1M) make 40k+ a non-issue, and
**prompt caching (coming — DAT-599, not designed-around) over a STABLE prefix sent in a loop
gives a very high cache-hit ratio** → the repetition is cheap. So the goal is **not** token
leanness — it is **never being blind**. Completeness + a cacheable layout + on-demand depth,
not a smaller context.

**The pattern.**
- **Complete map, never truncated.** One line per column/dim/measure WITH its size and a
  hint: `col : type, N distinct — [full set inlined | sample: v1, v2, v3 …]`. Low-card → the
  complete value-set inlined (uncapped). High-card → size + a small sample (never silent; the
  sample also lets the agent recognize opaque codes → abstain). Silence becomes information.
- **Serve the bounded full set; retrieve the unbounded rest.** Relevance is decided by the
  AGENT at reason-time, never pre-cut by us.
- **Batch + converge.** Where retrieval is needed it's one batched call for all named needs,
  and the loop terminates on completeness or fall-loud — never on a call-count wall.

**Engine GraphAgent (one-shot per metric, looped — stays one-shot per the agent-tier
boundary).** Its grounding space is BOUNDED (a vertical's concepts + a fact's low-card dims +
drivers), so serve it WHOLE: complete map + all low-card value-sets (uncapped) + all drivers,
laid out as a **stable cacheable prefix**; the metric spec + `prior_context` are the small
suffix. No relevance-gating, no metric-batching. A high-card column it would need but cannot
enumerate → **fall loud** (it can't retrieve — it is one-shot; the sample feeds the abstain).

**Cockpit answer agent (agentic).** Its space is OPEN-ENDED (any column/table, incl.
high-card), so: the same map with size+sample hints, plus a **batch `look_values([cols])`**
tool that returns the full value-set for named columns, each with its size/sample hint. The
agent reasons over the map, pulls what its question needs in one call, converges or falls
loud. (Not adding a tool to fish; the map makes the pull targeted.)

**Caps audit outcome.** Every silent cap on agent-context is removed or made non-silent:
`MAX_SLICES_PER_MEASURE` dropped (done, PR #374); `MAX_SECONDARY_PER_MEASURE` + the engine
`_VALUE_SET_RENDER_MAX` high-card suppression + `MAX_IDENTITIES_PER_TABLE` + `cycle.evidence[:3]`
→ become map-with-size/sample (non-silent) or get an overflow tail. Caps that already carry a
tail (`MAX_ENTITY_TABLES`, `MAX_DIMENSION_VALUES`) are fine.

**Spike: ground the size before setting any threshold (measure, don't guess).** Every
"inline vs sample" number here is a placeholder until measured — including the engine
`_VALUE_SET_RENDER_MAX` (currently 50) which must be replaced by a data-driven value. Method:
render the REAL `format_metadata_document` (with the new drivers/value-sets/cycle sections)
over an existing run in the live DB (no fresh smoke — the DB already holds a multi-table run)
and count tokens, broken down. Questions the spike must answer:
- **Total context tokens per metric** for a real multi-table run (the thing sent N times,
  cached). Confirm it sits comfortably inside the window with caching headroom.
- **Per-section breakdown** — where do the tokens go (map / value-sets / drivers / relationships
  / cycles / quality)? Which section dominates?
- **Value-set distribution** — how many low-card categoricals, the distinct-count histogram,
  the LARGEST value-set, and total value-set tokens. This sets the inline ceiling.
- **High-card tail** — how many columns exceed candidate thresholds (50 / 200 / 1000)? What
  would each threshold cost to inline (given caching)?
**Spike outcome (measured 2026-06-24, live 28-table run):**
- Full context = **52k chars ≈ 13k tokens** → size is a non-issue (caching + 1M window). The
  Tables section is 72%; Drivers tiny (6 measures, 134 tok).
- Renderable-categorical `distinct_count` (n=69) is **bimodal**: median **27**, then a gap,
  then a high-card tail (p90 ≈ 40k). Partial-rate is flat across 100–200 (40%) → the clusters
  are "genuine dims ≤ ~50" and "40k free-text/IDs", with clear air between.

**Decisions (the caps verdict):**
- **Drop ALL arbitrary display caps, in BOTH agents** — `MAX_SLICES` (done), `MAX_SECONDARY`,
  `MAX_IDENTITIES`, `MAX_DIMENSION_VALUES`, `MAX_SAMPLE`, `cycle.evidence[:3]`, and the engine
  `_VALUE_SET_RENDER_MAX` (50). They cap an already-bounded, structured, cached source — pure
  silent-recall downside. The win is formatting + a cache-stable layout, not capping.
- **One real knob: `top_k_values` for CATEGORICAL roles → a "reasonable top" ≈ 200** (vs 20
  today). The number is insensitive in [100,500] (it sits in the cluster gap); 200 makes every
  genuine dimension's value-set COMPLETE for the graph agent's baseline.
- **Gate by role AND cardinality:** a "categorical" with `distinct > reasonable-top` is not an
  aggregation partition (the 40k tail = free-text/high-card IDs that escaped the key/measure
  role) → serve **size + sample**, never enumerate. This is the dimension-vs-key boundary; it
  also patches a role-gating gap the spike exposed.
- **Division of labor:** the graph agent grounds the **baseline snippet library** on the
  reasonable-top complete value-sets (one-shot, can't drill). The cockpit `look_values` + the
  user **drill** the full low-card set (and the high-card tail) on demand. You don't enumerate
  a billion-value column for aggregation; the baseline covers what's aggregation-meaningful,
  the drill covers the rest.

**Build split.** This context model layers ON TOP of PR #374 as a follow-up: (a) unify the
map entry (size + inline-full-if-small / sample-if-large), (b) uncap the value-sets + make
high-card non-silent, (c) cache-friendly prefix layout in the GraphAgent prompt assembly,
(d) the cockpit `look_values([cols])` batch tool. #374 itself ships the feed + compose-CTE +
the one silent cap already dropped.

## 0. Spine + the decided calls (not open questions)

Context engine starves the two SQL authors of value-level grounding it already computes, and
the prompts tell them to improvise the row filter. Fix = SERVE the full feed to both, have
both author from a named-context/blueprint prompt, and unify how the engine composes/stores
SQL with how the answer agent already does it. Deterministic *evidence* + LLM *authoring* +
(later) human *teach*. No new artifact (DAT-620 cut), no deterministic builder, no value-judge.

Decided (so we stop re-litigating):
- **High-cardinality discriminator** (`distinct_count > K`, no complete value-set) → **fall loud**, accept the risk.
- **Verifier under CTE execution** → the composed `SELECT` **projects each step's scalar** as a column, so the kept support-gate still sees per-step values. One execution, floor intact.
- **Windowed / multi-row intermediate steps** (no single scalar to project — some `window` blueprints) → the per-step support gate fires **only on scalar extract steps**; multi-row steps are verified at the **composed output** only. This is the boundary that historically made the shape tail hard; we name it instead of pretending every step is scalar.
- **Asymmetric feed — producer vs consumer (NOT both-full).** The engine GraphAgent is the grounding PRODUCER (one-shot, writes the snippet library) → FULL feed. The cockpit answer agent is the CONSUMER (already searches the library via `snippet_search`, token-constrained, user-feedback loop) → LEAN feed: snippet prior-art + `column_mappings_basis` on results + slice-catalog `distinct_values` (the naturally bounded value-set) + the prompt rules. NO full `top_values`/driver pre-inject on the answer agent — that's the token graveyard. This dissolves the "relevance-gate" problem: the slice catalog IS the bounded set; no need to guess relevance.
- **Channel precedence is a GRAPH-AGENT prompt rule** (no binding artifact, 620 cut): meaning (concept vocab) selects which values; `top_values` is the COMPLETE set; drivers `interesting_slices` say which of those carry data/effect (a hint, never the set); human/teach > driver > top_values when they conflict. So the top-3 driver slices are never mistaken for the whole set. **Obsolete for the answer agent** — it gets no raw `top_values`; its value-set is the slice catalog.
- **"Never force-fit — abstain"** is an explicit system-prompt rule in BOTH prompts (eval-proven: it drove precision up and made the agent fail-correctly when a value isn't groundable, vs the ~11% force-fit baseline). This is the prompt-side of high-card → fall loud.
- **Engine mirrors the answer agent**: emit a snippet list AND a composed CTE, execute the CTE as ONE statement, classify-don't-re-execute. No CTE dissection.
- **Post-processing** = keep the cheap `verifier.py` floor, NO new sqlglot checker, ADD feedback loops.
- **Blueprint shape coverage** is real-LLM eval-gated (the gate so far only proved semantic names).

## 1. The feed — asymmetric: graph agent = grounding PRODUCER, answer agent = CONSUMER

The two agents have independent context paths (`reference_two_sql_agents_and_context`) AND
different ROLES in the grounding lifecycle, so they are NOT symmetric feed-targets:
- **Engine GraphAgent = PRODUCER.** One-shot at begin_session/operating_model; does the
  expensive value→concept grounding and **writes the snippet library** (durable concept→SQL +
  `column_mappings_basis`). Takes the **FULL feed** — a one-time investment that initializes
  the prior art. Context path: `graphs/context.py::format_metadata_document`.
- **Cockpit answer agent = CONSUMER.** Iterative, token-constrained both ways, exhaustion-prone
  (DAT-608), with the **user as a feedback loop**. It already searches the produced library
  (`snippet_search` — a TOOL; snippets are NOT pre-injected at start) and already has a
  pre-injected, capped `<drivers>` block. So it gets **LEAN** additions only — fire-hosing it
  raw value-sets is a token graveyard and redundant with the snippets/drivers it already has.
  Its 616 footprint = three small things (slice-catalog `distinct_values` into `<dimensions>`,
  `column_mappings_basis` on `snippet_search` results, the grounding/abstain prompt line).
  Context path: `query-context.ts` blocks. (Drivers-on-demand = a FUTURE DAT-573 question, not
  616; its exhaustion risk argues for keeping the capped pre-injected block over a tool.)

## 2. Engine: mirror the answer agent's compose/execute/store

Today (`query/execution.py`): `_execute_step` (`:147`) `CREATE TEMP VIEW step_id AS sql` per
step, `result[0]` per step (`:165`); `_execute_final` (`:204`) runs `final_sql` separately;
views cursor-scoped, never dropped (`:142`). The answer agent instead composes ONE CTE
(`composeStandalone`, `run-steps.ts:119-133`) and executes/validates it as a single statement.

- **2.1** Python mirror of `composeStandalone` in `execute_sql_steps` (`execution.py:65`): fold
  steps + `final_sql` into `WITH <step_id> AS (<sql>), … <final_sql>`, merging a leading `WITH`
  in `final_sql` exactly as the TS does. Engine scalar steps already alias `AS value` and
  `final_sql` references them via `(SELECT value FROM step_id)` — compatible.
- **2.2** Execute the composed statement ONCE; the outer `SELECT` also projects each **scalar**
  step's value (`…, (SELECT value FROM a) AS a, …`) so `StepResult.value` stays populated for the
  verifier. **Multi-row steps** (some `window` blueprints) have no scalar to project → they are
  NOT in the per-step gate; the composed output is verified instead (decided call §0).
  `_execute_step` + `_execute_final` temp-view path → DELETE.
- **2.3** Cached snippets become inline CTEs (never re-executed as views) ⇒ "don't re-execute
  stored snippets as views" falls out for free. `_save_snippets` (`agent.py:914`) /
  `_lookup_snippets` (`:1052`) UNCHANGED — per-step from the emitted steps, no CTE dissection.
- **2.4** Persist/return the composed CTE as the executable artifact (the engine's `grid.sql`
  equivalent), so a metric carries BOTH the snippet list AND the CTE.

## 3. Prompts — both, grounded + blueprinted

- **3.1 Engine `graph_sql_generation.yaml`:** grounding contract + fall-loud + generic
  placeholders — DONE (v4.0). NEW: mirror the answer-agent prompt's `<steps>`/`<reuse>` wording
  (`prompts/query.ts:50-57,40-48`) — standalone steps aliased `AS value`, `final_sql` references
  step CTEs by name, reuse a snippet faithfully. NEW: **blueprint LIBRARY** — a few-shot exemplar
  per shape class (simple aggregate / end-of-period latest-period CTE / window / multi-column
  conjunction / ratio). Shape coverage is taught, not hoped.
- **3.2 Cockpit `prompts/query.ts`:** add the value-grounding instruction — filter by the
  enumerated values from `<dimensions>` (slice-catalog `distinct_values`) and the reused
  snippet's `column_mappings_basis`, honor `exclude` patterns, fall loud / abstain when
  ungroundable. Mirror §3.1's grounding + §3.4 abstain rule. (No raw value-set pre-inject —
  consumer is lean per §0.)
- **3.3 Channel-precedence rule (GRAPH-AGENT prompt only):** concept vocab = meaning (which
  values); `top_values` = the COMPLETE set; driver `interesting_slices` = which of those carry
  data/effect (a HINT, never the set — the top-3 are not "all revenue values"); human/teach >
  driver > `top_values` on conflict. Without this the agent mistakes the salient slices for the
  whole set → undercount. Obsolete for the answer agent (no raw `top_values` there).
- **3.4 "Never force-fit — abstain" (BOTH prompts):** explicit rule — if a value can't be
  confidently mapped to a concept, abstain (fall loud) rather than assign it. Eval-proven
  precision lever (vs ~11% force-fit baseline); pairs with high-card → fall loud.
