# DAT-629 — Topo-warm the shared-node DAG before the metric fan-out

Epic: DAT-543. Refine approach: A+ (agreed 2026-06-24, captured on the DAT-629 ticket).

## Problem (recap)

`metrics_phase` authors metrics **in parallel** (`asyncio.gather`, sem 10). Each metric's
`agent.execute` calls `_lookup_snippets` at its start; siblings start ~together, so a shared
sub-node (e.g. the `cost_of_goods_sold` extract) is **cold** for everyone → each metric
independently LLM-authors it → divergence, some ground to an empty filter → born-loud
`no support`. Cross-run reuse works; within-run is racy. Every observed failure
(`COGS`/`revenue`/`current_assets`/`depreciation`/`operating_expense`) is an **EXTRACT**.

## Approach — warm, don't restructure

The snippet cache already covers all step types (`_lookup_snippets`/`_save_snippets`:
EXTRACT by `(standard_field,statement,aggregation)`, CONSTANT by param, FORMULA by
`find_by_expression`), and a fully-cached graph assembles with **no LLM**
(`_assemble_from_snippets`). So the cache mechanism is complete — the only defect is it's
cold during parallel authoring. Fix = **warm the unique nodes in dependency order before
the fan-out**; the existing per-metric `execute` then assembles from the warm cache —
unchanged, no race, consistent. `GraphStep.depends_on` is explicit; `networkx` is in-stack.

## Scope

**DO change**
- `pipeline/phases/metrics_phase.py` — insert a warming pre-pass between `compose` and `execute`.
- NEW `graphs/node_warming.py` (name TBD) — pure DAG build (unique-node keying + edges) +
  the per-node mini-graph warmer. Keep the pure parts unit-testable.
- `tests/unit/...` — DAG builder + warming + a two-metrics-share-COGS regression.

**DO NOT change**
- `graphs/agent.py` execute/lookup/save core (reuse as-is; a thin helper at most).
- `query/snippet_library.py` (the cache + keys are correct).
- The per-metric artifact lifecycle in `metrics_phase` (declare/compose/execute stays;
  warming is additive, artifacts stay per-metric — no remap).
- `query/execution.py` compose-CTE; cockpit; prompts.

## Phases

1. **DAG builder (pure, no execution).** From the parsed metric graphs, compute each step's
   global key (extract concept-key / constant param / formula expression), map
   `(graph_id, local_step_id) → global_key`, build a `nx.DiGraph` (edges from `depends_on`),
   `nx.find_cycle` → fail-loud, `nx.topological_generations` → waves. Unit tests: dedup
   across graphs, edge wiring, cycle detection, generation order.
2. **Node warmer.** Given a global node + its (already-warmed) deps, construct a minimal
   `TransformationGraph` (the node's representative step + transitive dep steps, re-wired to
   local step_ids, trivial `final_sql = SELECT value FROM <node>`) and run `agent.execute` →
   mints the concept-keyed snippet. Start with extracts (gen-0, no deps — trivial mini-graph);
   then formula/constant nodes (deps resolve from the warm cache). The snippet is concept-keyed,
   so the throwaway mini-graph step_ids don't matter to the metric's later lookup.
3. **Wire into `metrics_phase`.** After compose, run the warmer over the topo generations
   (parallel within a generation via the existing `asyncio.to_thread`+sem machinery; sequential
   across generations). Serial fallback when `ctx.manager is None`. Then the existing
   `_execute_metrics_*` runs unchanged — now cache-warm.
4. **Verify.** Unit: two metrics sharing `cost_of_goods_sold` both get the same extract SQL
   and both execute (no `no support`). Real: re-smoke the BookSQL run — `ebitda`/`net_income`
   that currently fall loud on extract `no support` should execute.

## Key risk / the one to watch

**Mini-graph construction fidelity (Phase 2)** — re-wiring `depends_on`/step_ids so a formula
node's mini-graph authors correctly with its deps served from cache. Extract warming (gen-0)
is trivial and fixes the observed bug; formula warming is the part that needs care. If formula
mini-graphs fight the agent, gen-0-only (extracts) still ships the bug fix — fall back to that
and note formula racing as a follow-on.

## Explicitly OUT of scope

Formula-string **consistency** (`operating_income` written identically across metrics) — a
config/vertical-definition validation concern (DAT-479/481 neighbourhood), not a runtime race.
A+ dedupes formulas only when the expression strings already match; it does not canonicalize.

## Test strategy

Unit (pure DAG builder; warming with a stub agent; the shared-COGS regression) + a real
BookSQL re-smoke for end-to-end confirmation. No eval handoff unless the response shape changes
(it doesn't — same artifacts, same snippets).
