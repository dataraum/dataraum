## Decisions
- fork: is truncation a contributor? · criterion: `stop_reason` distribution in the on-disk dumps
  · call: NO — zero `max_tokens` across 689 captured calls · evidence: dump scan 2026-07-20.
  Lead had said so before the data confirmed it.
- fork: does removing the forced tool silently enable thinking? · criterion: read the provider's
  thinking branch · call: NO — disabled explicitly by model prefix, independent of tool_choice
  · evidence: `anthropic.py:370-374` + `:81-85`.
- ruling (lead, 2026-07-20): **an optional field is usually a modelling mistake.** Per optional
  field, three dispositions, applied in the pre-flight before any migration code: nothing consumes
  it -> CUT; something consumes it -> REQUIRED with documented "" / [] sentinel; genuinely
  either/or -> both required with sentinel, NEVER a union (unions carry their own 16-per-request
  cap). Relayed to both lanes. Rationale: Pydantic/Zod render optionals as `anyOf` with null, so
  an optional plausibly spends from BOTH the 24-optional and 16-union budgets — cutting removes
  the pressure at the source instead of engineering around it. Out of scope, noted for later:
  optional free-text fields as a way to induce reasoning.
- fork: is `extract`/`formula`/`constant` a retired design? · criterion: engine dispatch sites +
  shipped YAML usage · call: NO, all three live — `agent.py:600` dispatches CONSTANT,
  `compose_formula_sql`/`compose_constant_sql` compose them, shipped YAML histogram is 44 extract /
  24 formula / 4 constant. Nuance the lead remembered correctly: DAT-643 made EXTRACT the SOLE LLM
  authoring surface (`agent.py:721`) — FORMULA/CONSTANT SQL is composed deterministically. The DAG
  still DECLARES them; only the SQL authoring moved. Strengthens the discriminated-union plan:
  each variant requires exactly what `compose_*_sql` reads.
- decision: `GraphStep.validations` (StepValidation, DAT-616) is MISSING from the cockpit mirror,
  so no induced/taught metric can declare a post-execution check while shipped ones can
  (`verifier.py:117` enforces them; dpo/dso/dio/current_ratio declare them). Consumed => disposition
  2: add as REQUIRED with documented `[]`. Zero cap cost (3 required subfields, no optionals, no
  unions). Induction prompt must ask for a check on the output step, else the field stays empty
  forever. IN SCOPE — the metrics lane rewrites this exact contract.
- ruling (lead, 2026-07-20): **no cache optimisation on an unstable base.** DAT-807 changed prompt
  text AND `output_config` together, so any cache hit-rate delta is unattributable. DAT-601
  (`graph_sql_generation` cacheability) is PARKED until the mechanism is proven stable — not
  absorbed here, not "fixed". `output_config.format` invalidating the thread's prompt cache is an
  expected consequence of the migration to re-measure later, NOT a regression. Reinforced by
  DAT-599's own principle: latency is dominated by OUTPUT tokens (serial decode), so caching is a
  cost lever and the lowest-value thing to chase while correctness is open.
- naming: `tool_repair.py` holds a SQL-provenance contract check, not a JSON repair — it parses the
  emitted SQL with DuckDB and verifies the declared `column_mappings_basis` against the served
  relation schema (`graphs/grounding_validation.py:53-110`). Once `repair_tool_output` is deleted
  the module is only that, so it gets renamed in the same edit.

## Decisions (cont.)
- fork: what actually drives the UNDOCUMENTED compiled-grammar size limit? · criterion: probe the
  live compiler with lever-variants of `InducedMetrics` (a schema-compile rejection is a
  request-time 400 — no tokens billed — so the real compiler is a free oracle) · call: **union
  BRANCHES, not bytes** · evidence: 5 step-shaped branches rejected; removing either union
  (→3) accepted; stripping EVERY description (11108 → 4091 bytes) still rejected. So prose is
  free and restating a shape is not.
- fork: which union to cut? · call: the structurally separate `output_step` (2 branches),
  keeping the 3-branch per-type dependency-step union · evidence: the per-type union is what
  guarantees each step carries exactly the fields its composer reads — that is semantic; the
  output step's separateness only bought guarantees that are cheap to re-express
  (`output_step_id` is a single string ⇒ exactly one; constant-as-output ⇒ a converter throw;
  the mandatory check ⇒ prompt + born-loud, which is where the shipped policy already put it).
- ruling: the guard covers EVERY `output_config.format` schema, not a sample. The four sites
  lane A migrated before the guard existed are exactly where the three 400s came from; a guard
  that covers 5 of 12 reproduces that gap by construction.
- correction: the earlier claim that ignored keywords (`minLength` etc.) still spend the
  grammar-size budget was speculation stated as fact. The probe disproves it — descriptions and
  ignored keywords cost nothing. The `WASTEFUL` list built on it is deleted.
