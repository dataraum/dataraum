# DAT-616 ‚Äî Feed both SQL agents the full grounding, and unify how they author/store SQL

Status: PLAN (full scope, 2026-06-24). The goal is not a minimal diff ‚Äî it is to make BOTH
SQL authors stop emitting silently-wrong metric SQL. We proved both are starved; this fixes
both. Wiring prompts + context is small surgery, not big scope ‚Äî so the plan goes wide on
purpose. File:line verified on `feat/dat-616-feed-grounding`.

Origin of the feed analysis: `dat616-serve-then-verify-design.md` ¬ß1‚Äì¬ß3 (the refined gap
inventory). This plan keeps that feed in full, drops its ¬ß4 verifier, and adds the
compose-CTE mirror + feedback loops decided 2026-06-24.

## 0. Spine + the decided calls (not open questions)

Context engine starves the two SQL authors of value-level grounding it already computes, and
the prompts tell them to improvise the row filter. Fix = SERVE the full feed to both, have
both author from a named-context/blueprint prompt, and unify how the engine composes/stores
SQL with how the answer agent already does it. Deterministic *evidence* + LLM *authoring* +
(later) human *teach*. No new artifact (DAT-620 cut), no deterministic builder, no value-judge.

Decided (so we stop re-litigating):
- **High-cardinality discriminator** (`distinct_count > K`, no complete value-set) ‚Üí **fall loud**, accept the risk.
- **Verifier under CTE execution** ‚Üí the composed `SELECT` **projects each step's scalar** as a column, so the kept support-gate still sees per-step values. One execution, floor intact.
- **Windowed / multi-row intermediate steps** (no single scalar to project ‚Äî some `window` blueprints) ‚Üí the per-step support gate fires **only on scalar extract steps**; multi-row steps are verified at the **composed output** only. This is the boundary that historically made the shape tail hard; we name it instead of pretending every step is scalar.
- **Asymmetric feed ‚Äî producer vs consumer (NOT both-full).** The engine GraphAgent is the grounding PRODUCER (one-shot, writes the snippet library) ‚Üí FULL feed. The cockpit answer agent is the CONSUMER (already searches the library via `snippet_search`, token-constrained, user-feedback loop) ‚Üí LEAN feed: snippet prior-art + `column_mappings_basis` on results + slice-catalog `distinct_values` (the naturally bounded value-set) + the prompt rules. NO full `top_values`/driver pre-inject on the answer agent ‚Äî that's the token graveyard. This dissolves the "relevance-gate" problem: the slice catalog IS the bounded set; no need to guess relevance.
- **Channel precedence is a GRAPH-AGENT prompt rule** (no binding artifact, 620 cut): meaning (concept vocab) selects which values; `top_values` is the COMPLETE set; drivers `interesting_slices` say which of those carry data/effect (a hint, never the set); human/teach > driver > top_values when they conflict. So the top-3 driver slices are never mistaken for the whole set. **Obsolete for the answer agent** ‚Äî it gets no raw `top_values`; its value-set is the slice catalog.
- **"Never force-fit ‚Äî abstain"** is an explicit system-prompt rule in BOTH prompts (eval-proven: it drove precision up and made the agent fail-correctly when a value isn't groundable, vs the ~11% force-fit baseline). This is the prompt-side of high-card ‚Üí fall loud.
- **Engine mirrors the answer agent**: emit a snippet list AND a composed CTE, execute the CTE as ONE statement, classify-don't-re-execute. No CTE dissection.
- **Post-processing** = keep the cheap `verifier.py` floor, NO new sqlglot checker, ADD feedback loops.
- **Blueprint shape coverage** is real-LLM eval-gated (the gate so far only proved semantic names).

## 1. The feed ‚Äî asymmetric: graph agent = grounding PRODUCER, answer agent = CONSUMER

The two agents have independent context paths (`reference_two_sql_agents_and_context`) AND
different ROLES in the grounding lifecycle, so they are NOT symmetric feed-targets:
- **Engine GraphAgent = PRODUCER.** One-shot at begin_session/operating_model; does the
  expensive value‚Üíconcept grounding and **writes the snippet library** (durable concept‚ÜíSQL +
  `column_mappings_basis`). Takes the **FULL feed** ‚Äî a one-time investment that initializes
  the prior art. Context path: `graphs/context.py::format_metadata_document`.
- **Cockpit answer agent = CONSUMER.** Iterative, token-constrained both ways, exhaustion-prone
  (DAT-608), with the **user as a feedback loop**. It already searches the produced library
  (`snippet_search` ‚Äî a TOOL; snippets are NOT pre-injected at start) and already has a
  pre-injected, capped `<drivers>` block. So it gets **LEAN** additions only ‚Äî fire-hosing it
  raw value-sets is a token graveyard and redundant with the snippets/drivers it already has.
  Its 616 footprint = three small things (slice-catalog `distinct_values` into `<dimensions>`,
  `column_mappings_basis` on `snippet_search` results, the grounding/abstain prompt line).
  Context path: `query-context.ts` blocks. (Drivers-on-demand = a FUTURE DAT-573 question, not
  616; its exhaustion risk argues for keeping the capped pre-injected block over a tool.)

## 2. Engine: mirror the answer agent's compose/execute/store

Today (`query/execution.py`): `_execute_step` (`:147`) `CREATE TEMP VIEW step_id AS sql` per
step, `result[0]` per step (`:165`); `_execute_final` (`:204`) runs `final_sql` separately;
views cursor-scoped, never dropped (`:142`). The answer agent instead composes ONE CTE
(`composeStandalone`, `run-steps.ts:119-133`) and executes/validates it as a single statement.

- **2.1** Python mirror of `composeStandalone` in `execute_sql_steps` (`execution.py:65`): fold
  steps + `final_sql` into `WITH <step_id> AS (<sql>), ‚Ä¶ <final_sql>`, merging a leading `WITH`
  in `final_sql` exactly as the TS does. Engine scalar steps already alias `AS value` and
  `final_sql` references them via `(SELECT value FROM step_id)` ‚Äî compatible.
- **2.2** Execute the composed statement ONCE; the outer `SELECT` also projects each **scalar**
  step's value (`‚Ä¶, (SELECT value FROM a) AS a, ‚Ä¶`) so `StepResult.value` stays populated for the
  verifier. **Multi-row steps** (some `window` blueprints) have no scalar to project ‚Üí they are
  NOT in the per-step gate; the composed output is verified instead (decided call ¬ß0).
  `_execute_step` + `_execute_final` temp-view path ‚Üí DELETE.
- **2.3** Cached snippets become inline CTEs (never re-executed as views) ‚áí "don't re-execute
  stored snippets as views" falls out for free. `_save_snippets` (`agent.py:914`) /
  `_lookup_snippets` (`:1052`) UNCHANGED ‚Äî per-step from the emitted steps, no CTE dissection.
- **2.4** Persist/return the composed CTE as the executable artifact (the engine's `grid.sql`
  equivalent), so a metric carries BOTH the snippet list AND the CTE.

## 3. Prompts ‚Äî both, grounded + blueprinted

- **3.1 Engine `graph_sql_generation.yaml`:** grounding contract + fall-loud + generic
  placeholders ‚Äî DONE (v4.0). NEW: mirror the answer-agent prompt's `<steps>`/`<reuse>` wording
  (`prompts/query.ts:50-57,40-48`) ‚Äî standalone steps aliased `AS value`, `final_sql` references
  step CTEs by name, reuse a snippet faithfully. NEW: **blueprint LIBRARY** ‚Äî a few-shot exemplar
  per shape class (simple aggregate / end-of-period latest-period CTE / window / multi-column
  conjunction / ratio). Shape coverage is taught, not hoped.
- **3.2 Cockpit `prompts/query.ts`:** add the value-grounding instruction ‚Äî filter by the
  enumerated values from `<dimensions>` (slice-catalog `distinct_values`) and the reused
  snippet's `column_mappings_basis`, honor `exclude` patterns, fall loud / abstain when
  ungroundable. Mirror ¬ß3.1's grounding + ¬ß3.4 abstain rule. (No raw value-set pre-inject ‚Äî
  consumer is lean per ¬ß0.)
- **3.3 Channel-precedence rule (GRAPH-AGENT prompt only):** concept vocab = meaning (which
  values); `top_values` = the COMPLETE set; driver `interesting_slices` = which of those carry
  data/effect (a HINT, never the set ‚Äî the top-3 are not "all revenue values"); human/teach >
  driver > `top_values` on conflict. Without this the agent mistakes the salient slices for the
  whole set ‚Üí undercount. Obsolete for the answer agent (no raw `top_values` there).
- **3.4 "Never force-fit ‚Äî abstain" (BOTH prompts):** explicit rule ‚Äî if a value can't be
  confidently mapped to a concept, abstain (fall loud) rather than assign it. Eval-proven
  precision lever (vs ~11% force-fit baseline); pairs with high-card ‚Üí fall loud.
